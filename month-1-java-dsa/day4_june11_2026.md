## DSA — HashMap Patterns

### 1. Valid Anagram — LC#242
**Pattern:** HashMap frequency count → optimised to `int[26]`

#### Brute Force — O(n log n)
```java
class Solution {
    public boolean isAnagram(String s, String t) {
        if (s.length() != t.length()) return false;
        char[] sChar = s.toCharArray();
        Arrays.sort(sChar);
        char[] tChar = t.toCharArray();
        Arrays.sort(tChar);
        for (int i = 0; i < s.length(); i++) {
            if (sChar[i] != tChar[i]) return false;
        }
        return true;
    }
}
// Runtime: sort-based, O(n log n) time, O(n) space
// Tip: final loop can be replaced with Arrays.equals(sChar, tChar)
```

#### Optimal — O(n) | 2ms | Beats 99.61%
```java
class Solution {
    public boolean isAnagram(String s, String t) {
        if (s.length() != t.length()) return false;
        int[] freq = new int[26];
        for (char c : s.toCharArray()) freq[c - 'a']++;
        for (char c : t.toCharArray()) freq[c - 'a']--;
        for (int f : freq) if (f != 0) return false;
        return true;
    }
}
// Key insight: bounded characters → int[26] beats HashMap (no boxing overhead)
// c - 'a' maps 'a'→0, 'b'→1 ... 'z'→25
```

### 2. Group Anagrams — LC#49
**Pattern:** HashMap with sorted string as key

#### Solution — O(n * k log k) | 6ms | Beats 99.43%
```java
class Solution {
    public List<List<String>> groupAnagrams(String[] strs) {
        HashMap<String, List<String>> map = new HashMap<>();
        for (String str : strs) {
            char[] c = str.toCharArray();
            Arrays.sort(c);
            String sortedStr = new String(c);
            map.computeIfAbsent(sortedStr, k -> new ArrayList<>()).add(str);
        }
        return new ArrayList<>(map.values());
    }
}
// Key insight: char[] cannot be a HashMap key (reference equality, not content)
//              always convert to String after sorting
// computeIfAbsent: creates list if absent, returns existing if present — cleaner than containsKey
```

### 3. Top K Frequent Elements — LC#347
**Pattern:** HashMap + bucket sort (optimal) or min-heap

#### Approach 1 — Min-Heap O(n log k) | 14ms | Beats 70.48%
```java
class Solution {
    public int[] topKFrequent(int[] nums, int k) {
        HashMap<Integer, Integer> map = new HashMap<>();
        for (int num : nums) {
            map.put(num, map.getOrDefault(num, 0) + 1);
        }
        // min-heap ordered by frequency — int[] is {num, frequency}
        PriorityQueue<int[]> minHeap = new PriorityQueue<>((a, b) -> a[1] - b[1]);
        for (Map.Entry<Integer, Integer> entry : map.entrySet()) {
            minHeap.add(new int[]{entry.getKey(), entry.getValue()});
            if (minHeap.size() > k) minHeap.poll(); // evict least frequent
        }
        int[] result = new int[minHeap.size()];
        int index = 0;
        while (!minHeap.isEmpty()) {
            result[index++] = minHeap.poll()[0];
        }
        return result;
    }
}
// Key insight: keep heap size at k, always evict minimum → only top k survive
// poll() returns int[] — extract number via [0]
```

#### Approach 2 — Bucket Sort O(n) | 11ms | Beats 96.07%
```java
class Solution {
    public int[] topKFrequent(int[] nums, int k) {
        HashMap<Integer, Integer> map = new HashMap<>();
        for (int num : nums) {
            map.put(num, map.getOrDefault(num, 0) + 1);
        }
        // bucket index = frequency, max frequency cannot exceed nums.length
        List<Integer>[] buckets = new List[nums.length + 1];
        for (Map.Entry<Integer, Integer> entry : map.entrySet()) {
            int freq = entry.getValue();
            if (buckets[freq] == null) buckets[freq] = new ArrayList<>();
            buckets[freq].add(entry.getKey());
        }
        int[] result = new int[k];
        int index = nums.length;
        int filled = 0;
        while (index >= 0 && filled < k) {
            if (buckets[index] != null) {
                for (int num : buckets[index]) {
                    if (filled < k) result[filled++] = num;
                }
            }
            index--;
        }
        return result;
    }
}
// Key insight: frequency range is bounded by n → bucket sort wins over heap
// Read buckets right to left (highest freq first) until k elements collected
```

## DSA — Stack Patterns

### 4. Valid Parentheses — LC#20
**Pattern:** Stack + HashMap for bracket matching

#### Brute Force → same as optimal for this problem (stack is the natural fit)

#### Optimal — O(n) | 3ms | Beats 87.87%
```java
class Solution {
    public boolean isValid(String s) {
        Deque<Character> stack = new ArrayDeque<>();
        HashMap<Character, Character> map = new HashMap<>();
        map.put(')', '(');
        map.put('}', '{');
        map.put(']', '[');
        for (char c : s.toCharArray()) {
            switch (c) {
                case '(':
                case '{':
                case '[':
                    stack.push(c);
                    break;
                case ')':
                case '}':
                case ']':
                    if (stack.isEmpty() || stack.pop() != map.get(c)) return false;
            }
        }
        return stack.isEmpty();
    }
}
// Key insight: map closing→opening bracket
//              always check isEmpty() before pop() to avoid exception
//              end condition: stack must be empty (no unmatched opens)
```

### 5. Min Stack — LC#155
**Pattern:** Two stacks — main stack + parallel min tracking stack

#### Optimal — O(1) all operations | 32ms
```java
class MinStack {
    Deque<Integer> mainStack = new ArrayDeque<>();
    Deque<Integer> minStack = new ArrayDeque<>();

    public MinStack() {}

    public void push(int val) {
        mainStack.push(val);
        int curMin = minStack.isEmpty() ? val : Math.min(val, minStack.peek());
        minStack.push(curMin);
    }

    public void pop() {
        mainStack.pop();
        minStack.pop(); // always pop both together
    }

    public int top() {
        return mainStack.peek();
    }

    public int getMin() {
        return minStack.peek(); // O(1) — current min always at top
    }
}
// Key insight: minStack stores the running minimum AT EACH POINT IN TIME
//              when you pop main, pop min too — previous min is restored automatically
// Example:
//   push(-2): main=[-2]        min=[-2]
//   push(0):  main=[-2,0]      min=[-2,-2]
//   push(-3): main=[-2,0,-3]   min=[-2,-2,-3]
//   pop():    main=[-2,0]       min=[-2,-2]   → getMin() = -2 ✅
```

### 6. Daily Temperatures — LC#739
**Pattern:** Monotonic stack (next greater element)

#### Brute Force — O(n²) | TLE on large input
```java
class Solution {
    public int[] dailyTemperatures(int[] temperatures) {
        int[] result = new int[temperatures.length];
        for (int i = 0; i < temperatures.length; i++) {
            for (int j = i + 1; j < temperatures.length; j++) {
                if (temperatures[j] > temperatures[i]) {
                    result[i] = j - i;
                    break;
                }
            }
        }
        return result;
    }
}
// Fails: TLE on 47/48 test cases — O(n²) too slow for large arrays
```

#### Optimal — Monotonic Stack O(n) | 24ms | Beats 86.88%
```java
class Solution {
    public int[] dailyTemperatures(int[] temperatures) {
        int[] result = new int[temperatures.length];
        Deque<Integer> stack = new ArrayDeque<>(); // stores indices, not values
        for (int i = 0; i < temperatures.length; i++) {
            // resolve all indices waiting for a warmer day
            while (!stack.isEmpty() && temperatures[i] > temperatures[stack.peek()]) {
                int idx = stack.pop();
                result[idx] = i - idx;
            }
            stack.push(i);
        }
        return result;
    }
}
// Key insight: store INDICES not values — answer is always i - idx
// Monotonic stack rule: next greater/smaller element for every index → O(n)
// Unresolved indices left in stack → result stays 0 (default) ✅
```

## AWS — Key Concepts Covered

### RDS
- Multi-AZ: synchronous replication, standby NOT readable, auto failover ~60-120s, same endpoint
- Read Replicas: asynchronous, ARE readable, can be cross-region, manual promotion
- Parameter Groups: always create custom (default non-editable), static=reboot, dynamic=immediate
- Backups: restore always creates NEW instance
- Security: private subnet, KMS encryption must be set at creation

### SQS + SNS
- SQS: pull model, one consumer per message, persists up to 14 days
- SNS: push model, fan-out to multiple subscribers (SQS, Lambda, email, HTTP)
- Fan-out pattern: SNS topic → multiple SQS queues
- Hands-on completed: SNS topic → email subscriber → message published → email received ✅

## Interview Q&A

### HashMap / DSA

**Q: When would you use `int[]` over `HashMap<Character, Integer>` for frequency counting?**
A: When the input is bounded characters (e.g. lowercase English letters a-z), use `int[26]` with `c - 'a'` as index. Avoids boxing overhead (char→Character, int→Integer), direct array access is O(1) with no hashing. Result: 99%+ runtime vs ~30% with HashMap.

**Q: Why can't `char[]` be used as a HashMap key in Java?**
A: HashMap uses `.equals()` for key comparison. Two different `char[]` arrays with identical contents are not equal — Java compares references, not content. Always convert to `String` via `new String(charArray)` after sorting.

**Q: What is the time complexity of Group Anagrams and why?**
A: O(n * k log k) where n = number of strings, k = max string length. Sorting each string costs O(k log k), done for all n strings. Space: O(n * k) for the HashMap.

**Q: Explain the two-stack approach for Min Stack. Why not just track a single minVal variable?**
A: A single variable breaks when you pop the current minimum — you lose the previous minimum. The two-stack approach pushes the running minimum alongside every value. When you pop both stacks together, the previous minimum is automatically restored at the top of the min stack.

**Q: What is a monotonic stack and when do you use it?**
A: A stack that maintains elements in monotonically increasing or decreasing order. Use it for "next greater element", "next smaller element", or "previous greater/smaller element" problems. Achieves O(n) by ensuring each element is pushed and popped at most once.

### AWS RDS

**Q: What is the difference between Multi-AZ and Read Replicas?**
A: Multi-AZ = synchronous replication to a standby for high availability — standby is NOT readable, failover is automatic (~60-120s), same endpoint. Read Replicas = asynchronous replication for read scaling — ARE readable, promoted manually, can be cross-region.

**Q: RDS is in a private subnet and needs to download a patch. How does it reach the internet?**
A: Route table in the private subnet routes 0.0.0.0/0 → NAT Gateway (in public subnet) → IGW → internet. The RDS instance never has a public IP.

**Q: You restore an RDS backup. Where does it restore to?**
A: Always creates a NEW RDS instance — it does not restore in-place. You need to update your application's connection string to point to the new endpoint.

**Q: What happens to automated backups when you delete an RDS instance?**
A: Automated backups are deleted. Manual snapshots persist. Always take a final manual snapshot before deletion.

### AWS SQS + SNS

**Q: What is the fan-out pattern and how do you implement it in AWS?**
A: One event needs to trigger multiple independent consumers. Implement via SNS topic with multiple SQS queues as subscribers. Producer publishes once to SNS → SNS pushes to all subscribed SQS queues → each consumer processes independently and at its own pace.

**Q: What is SQS visibility timeout and why does it matter?**
A: When a consumer receives a message, it becomes invisible to other consumers for the visibility timeout duration. If the consumer processes and deletes it — done. If it fails and doesn't delete it, the message reappears after timeout for retry. Set timeout > your max processing time to avoid duplicate processing.

**Q: SQS vs SNS — when do you use each?**
A: SQS when you need reliable async job processing with one consumer per message, message persistence, and retry logic. SNS when you need to broadcast one event to multiple subscribers simultaneously (fan-out). Often used together: SNS for fan-out → SQS for reliable per-consumer processing.

**Q: What is a Dead Letter Queue (DLQ)?**
A: A separate SQS queue where messages are sent after exceeding the maximum receive count (failed processing attempts). Used to isolate poison-pill messages for debugging without blocking the main queue.
