## Slot 1 ‚Äî AWS IAM Deep Dive

### Policy Types
- **Identity-based policies**: attached to users/groups/roles ‚Äî define what the identity can do.
- **Resource-based policies**: attached to resources (S3, SQS, SNS, KMS) ‚Äî define who can access the resource. Only these can grant cross-account access directly.
- **Permission boundaries**: managed policy on a user/role setting the *maximum* possible permissions ‚Äî a ceiling, not a grant.
- **SCPs (Service Control Policies)**: AWS Organizations level ‚Äî restrict entire accounts/OUs, ceiling even for root.
- **Session policies**: passed at `AssumeRole` time ‚Äî further restrict permissions for that session only.

### Policy Evaluation Order
```
Explicit DENY anywhere ‚Üí DENY (always wins)
        ‚Üì
SCP allows it? ‚Üí No ‚Üí DENY
        ‚Üì Yes
Resource-based policy allows? ‚Üí Yes ‚Üí ALLOW (cross-account bypass)
        ‚Üì No
Permission boundary allows? ‚Üí No ‚Üí DENY
        ‚Üì Yes
Identity-based policy allows? ‚Üí No ‚Üí DENY
        ‚Üì Yes
Session policy allows? ‚Üí No ‚Üí DENY
        ‚Üì Yes
ALLOW
```
**Interview framing**: Effective permission = intersection of SCP ‚à© Boundary ‚à© Identity (‚à™ Resource for cross-account) ‚à© Session, minus any explicit deny.

### Trust Policy vs Permission Policy (Roles)
- **Trust policy**: WHO can assume this role (principal allowed to call `sts:AssumeRole`).
- **Permission policy**: WHAT the role can do once assumed.
- Common mistake: trying to restrict role usage via permission policy ‚Äî wrong, that's the trust policy's job.

### STS AssumeRole Flow
1. Principal A's identity policy must allow `sts:AssumeRole` on Role X.
2. Role X's trust policy must list Principal A as trusted.
3. A calls `sts:AssumeRole(RoleX)`.
4. STS validates both checks (AND, not OR).
5. STS issues temporary credentials (default 1hr, max per `MaxSessionDuration`, up to 12h).
6. A acts as Role X, bounded by X's permission policy (+ optional session policy).

### Cross-Account Access Pattern
- Standard: Account A's identity assumes a role in Account B (B's role trust policy trusts A, permission policy defines B-side access). Only temporary STS tokens cross the boundary ‚Äî no long-lived credentials.
- **ExternalId**: condition in trust policy to prevent "confused deputy" problem (e.g., SaaS vendors assuming roles into customer accounts).
- Alternative: resource-based policy on the target resource (S3, SQS, SNS, KMS, Lambda) can directly grant access without any role/AssumeRole ‚Äî but NOT for EC2 (no resource-based policy support).

### Interview Q&A
1. **Identity policy allows S3, OU-level SCP denies S3 ‚Äî outcome?** Denied. SCP is an account-wide ceiling, overrides any identity allow.
2. **Permission boundary vs SCP?** Boundary = single user/role within an account (delegation control). SCP = entire accounts/OUs via Organizations. Both are ceilings, neither grants.
3. **Lambda needs cross-account S3 write ‚Äî no access keys?** Either (a) Lambda execution role assumes a cross-account role (target trust policy trusts Lambda's role/account), or (b) target bucket's resource-based policy directly grants the Lambda execution role's ARN.
4. **Confused deputy problem + ExternalId fix?** Third-party role in your account could be tricked by an attacker who also accesses that third party. ExternalId is a shared-secret trust policy condition scoping the relationship.
5. **Can resource-based policy grant access with zero identity policy involved?** Yes ‚Äî for S3/SQS/SNS/KMS/Lambda, resource-based policy alone can grant a principal in another account access. Exception to "both sides must allow."

## Slot 2 ‚Äî DSA: Binary Search Patterns

| Problem | LC# | Result |
|---|---|---|
| Binary Search | 704 | 0ms, Beats 100% |
| Search Insert Position | 35 | 0ms, Beats 100% |
| Find Minimum in Rotated Sorted Array | 153 | 0ms, Beats 100% |

### Pattern Notes
- **LC#704 (exact match)**: `lo <= hi`, `mid = lo + (hi-lo)/2` (overflow-safe), update `lo=mid+1`/`hi=mid-1`.
- **LC#35 (boundary/insertion)**: identical code to #704, returns `lo` on not-found ‚Äî `lo` lands at the leftmost valid insertion point. Foundation for "lower_bound" style problems.
- **LC#153 (converge on rotated array)**: `lo < hi` (not `<=`), compare `nums[mid]` vs `nums[hi]` to determine sorted half, `hi = mid` (never `mid-1` ‚Äî mid could be the answer).

### Pattern Summary Table

| Variant | Loop condition | Update | Use case |
|---|---|---|---|
| Exact match | `lo <= hi` | `lo=mid+1` / `hi=mid-1` | find target index |
| Boundary/insertion | `lo <= hi` | same, return `lo` | first position where condition flips |
| Converge to element | `lo < hi` | `lo=mid+1` / `hi=mid` | min/max under rotation, property-change point |

**Interview framing**: "Sorted array + O(log n)" ‚Üí ask "exact value or boundary/property change?" ‚Äî determines which variant to apply.

## Slot 3 ‚Äî AWS Lambda + ECS Overview (no labs, overview only)

### Lambda
- Event-driven triggers: API Gateway, S3, SQS, SNS, EventBridge, DynamoDB Streams, ALB.
- Execution role defines AWS resource access at runtime.
- Cold start vs warm start; Provisioned Concurrency pre-warms environments.
- Concurrency: each concurrent invocation = separate execution environment; reserved vs unreserved pool.
- Timeout max 15 min; memory 128MB‚Äì10GB (CPU scales with memory).
- Layers: shared code/dependencies across functions.
- Event source mapping (SQS/Kinesis/DynamoDB Streams) = sync polling; S3/SNS = async invocation with retries + DLQ.
- VPC Lambda needs ENIs for private subnet resource access (e.g., RDS).

### ECS
- **Launch types**: EC2 (self-managed capacity) vs Fargate (serverless, pay per task).
- **Task Definition**: blueprint ‚Äî container image(s), CPU/memory, ports, env vars, IAM roles.
- **Task Role vs Execution Role**: Execution role = ECS agent pulls images/writes logs. Task role = app code calls AWS APIs (mirrors Lambda's execution role concept).
- **Service**: maintains desired task count, integrates with ALB/NLB, rolling deployments.
- **Cluster**: logical grouping of tasks/services (+ EC2 instances if EC2 launch type).

### Decision Framework
- Lambda: event-driven, short-lived, stateless.
- ECS+Fargate: long-running containerized microservices, no infra management.
- ECS+EC2: long-running + need infra control (GPU, cost optimization at scale, custom AMIs).

**Pointers used**: Stephane Maarek CCP course (Lambda + ECS/Fargate sections), AWS Skill Builder free modules, AWS docs on Task IAM role vs Task execution IAM role.

## Slot 4 ‚Äî DSA: Linked List Fast/Slow Pointer

| Problem | LC# | Result |
|---|---|---|
| Linked List Cycle | 141 | 0ms, Beats 100% |
| Middle of the Linked List | 876 | 0ms, Beats 100% |
| Linked List Cycle II | 142 | 0ms, Beats 100% |

### Pattern Notes
- **LC#141**: fast=2x slow; if cycle exists, fast laps slow within one full cycle loop. Check `fast != null && fast.next != null` before `fast.next.next`.
- **LC#876**: same loop structure; for even-length lists, `slow` naturally lands on the **second** middle node.
- **LC#142 (cycle start)**: two-phase ‚Äî (1) detect meeting point via fast/slow, (2) reset `slow = head`, advance both by 1 until they meet again ‚Üí meeting point = cycle start.
  - **Math**: `a` = head‚Üícycle-start distance, `b` = cycle-start‚Üímeeting-point, `c` = meeting-point‚Üícycle-start (remaining loop). Since fast travels 2x slow's distance: `a = (k-1)(b+c) + c`, meaning distance from head equals distance from meeting point to cycle start (mod cycle length).

### Pattern Summary Table

| Variant | Technique | Use case |
|---|---|---|
| Cycle detection (141) | fast=2x slow, check meeting | does a cycle exist? |
| Middle finding (876) | fast=2x slow, same loop | find midpoint, no second pass |
| Cycle start (142) | detect + reset-and-reconverge | find WHERE cycle begins |

**Interview framing**: "Linked list + no extra space" ‚Üí fast/slow pointer is the default first instinct for midpoint, cycle, or kth-from-end problems.

## Java ‚Äî JVM Internals: Heap Regions, GC, Class Loading

### Heap Regions
```
Heap
‚îú‚îÄ‚îÄ Young Generation
‚îÇ   ‚îú‚îÄ‚îÄ Eden          (new object allocation)
‚îÇ   ‚îú‚îÄ‚îÄ Survivor 0
‚îÇ   ‚îî‚îÄ‚îÄ Survivor 1     (objects ping-pong S0‚ÜîS1 across minor GCs)
‚îî‚îÄ‚îÄ Old Generation (Tenured)  (promoted long-lived objects)

Metaspace (off-heap) ‚Äî class metadata, replaced PermGen since Java 8
```
- Promotion to Old Gen governed by age threshold (`-XX:MaxTenuringThreshold`).
- Metaspace grows dynamically; unbounded class loading (classloader leaks) can OOM here.

### Garbage Collection

**Minor GC**: Eden fills ‚Üí live objects copied to Survivor (or promoted to Old). Fast, frequent.

**Major/Full GC**: Old Gen fills ‚Üí expensive, can cause noticeable stop-the-world (STW) pauses. Avoid manual `System.gc()`.

**Collectors**:

| Collector | Characteristics | Use case |
|---|---|---|
| Serial GC | Single-threaded, STW | Small heaps, single-core |
| Parallel GC | Multi-threaded, STW | Throughput-focused, batch |
| G1 (default since Java 9) | Region-based, concurrent + incremental, pause-time goals | General-purpose, low-latency |
| ZGC | Concurrent, sub-ms pauses, scales to TBs | Latency-critical, very large heaps |
| Shenandoah | Concurrent compaction, low pause | Alternative to ZGC (Red Hat) |

**G1 specifics**: heap divided into many equal-sized regions, roles assigned dynamically (not fixed Eden/Survivor/Old layout). Concurrent marking identifies garbage-heavy regions, collects those first. `-XX:MaxGCPauseMillis` sets a *goal*, not a guarantee.

**STW**: app threads pause during certain GC phases ‚Äî minimizing STW duration/frequency is the central GC tuning goal.

### Class Loading

**ClassLoader Hierarchy (delegation model)**:
```
Bootstrap ClassLoader (java.lang.*, core JDK ‚Äî native code)
   ‚Üì
Platform/Extension ClassLoader (JDK extensions)
   ‚Üì
Application/System ClassLoader (your app classpath)
   ‚Üì
Custom ClassLoaders (plugins, hot-reload)
```
- Delegation: requests go UP to parent first; child only loads if parent can't. Prevents app code from overriding core classes.

**Class Loading Phases**:
1. **Loading**: locate `.class` bytes, create `Class` object.
2. **Linking**:
   - Verification: bytecode safety/structure checks.
   - Preparation: allocate memory for static fields, set defaults (0/null/false).
   - Resolution: symbolic references ‚Üí direct references (can be lazy).
3. **Initialization**: static initializers + static field assignments run lazily, on first active use.

### Interview Q&A
1. **`new MyObject()` under memory pressure ‚Äî walk through it**: Allocated in Eden ‚Üí if full, minor GC copies live objects to Survivor ‚Üí repeated survival promotes to Old Gen ‚Üí if Old Gen fills, major GC (possibly STW) runs.
2. **Why G1 default, when to switch to ZGC?** G1 balances throughput/pause-time for most workloads with configurable goals. Switch to ZGC for very large heaps (10s‚Äì100s GB) or strict sub-ms latency SLAs.
3. **`NoClassDefFoundError` despite class on classpath ‚Äî debug approach?** Check if the class's static initializer threw on first load (causes `ExceptionInInitializerError` then `NoClassDefFoundError` on subsequent refs). Also check classloader visibility in modular/app-server setups.
4. **`ClassNotFoundException` vs `NoClassDefFoundError`?** Former: explicit load attempt (`Class.forName()`) failed, checked exception. Latter: class loaded successfully once but missing/failed on a later reference.
